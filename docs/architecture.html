1 개요

단일 FastAPI 프로세스가 REST API, 정적 SPA, 관리자 콘솔, WebSocket 실시간 시세, 내부 배치 트리거를 모두 담당한다. 영속성은 단일 SQLite(cache.db)가 분석 캐시·사용자/세션·포트폴리오·NAV 스냅샷·AI 사용량·공시/리포트 요약·운영 이벤트를 저장한다. 운영 작업(일일정산·장중 스냅샷·NPS·wiki/DART ingestion)은 systemd 타이머가 내부 엔드포인트 또는 CLI로 호출한다.

웹/실시간

FastAPI · Uvicorn

SSE 스트리밍 + /ws/quotes WebSocket

영속성

SQLite (WAL, busy_timeout)

KIS(국내/해외·WS), 네이버, yfinance, Upbit, DART, OpenRouter(AI)

배포

GitHub Actions → 자가호스팅 러너(홈 라즈베리파이)

master push = 자동 배포

3 런타임 · 배포

부팅 (core.lifespan)

  • load_environment() → .env → .env.<profile> → .kis.env → keys.txt 순으로 설정 로드
  • KIS 키 로드, KIS 프록시/close price 클라이언트 init, cache.init_db(), corp_codes 갱신 점검
  • 백그라운드 루프: watchdog(sd_notify), wiki ingestion, DART 리뷰, observability prune, (옵션) insight 워밍업
  • 종료 시 백그라운드 태스크 취소·대기, WS 정리, 클라이언트/DB close

배포 파이프라인

git push origin master └─▶ GitHub Actions: deploy-home.yml (self-hosted runner @ 홈 라즈베리파이) └─▶ deploy/deploy.sh → 코드 갱신 + 서비스 재시작 (≈30s)

과거 deploy-pages.yml(GitHub Pages 프론트 호스팅)은 프론트를 FastAPI가 직접 서빙하도록 전환하며 제거됨. 현재 프론트·API 모두 동일 출처(라즈베리파이)에서 서빙.

5 포트폴리오 서비스 분해 (services/portfolio)

라우터의 god-module화를 막기 위해 순수 로직을 도메인별로 추출 중이다. 의존 방향은 routes → services → (fx → currencies) 단방향.

identifiers

코드 정규화, 국내/우선주/현금성/특수자산 판정, 정적 해외 ticker

currencies ✦

통화·국적·FX코드 매핑 단일화 (기존 9곳 분산 → 1곳, 비대칭 문서화)

fx ✦

환율 조회/일별변동/원화 변환 + 캐시

special_assets ✦

KRX 금현물(네이버), 크립토(Upbit) 시세

foreign ✦

해외 시세(KIS 해외→yfinance→네이버 world), 이름/Reuters/ticker 해석, ticker map, yfinance 음성 캐시

benchmarks

기본 벤치마크, 표시명, 지표 등락률 변환, quote 캐시

dividends

배당 워밍업 대상 선정, 우선주-본주 동시 예열, TTL/중복 방지

history

NAV/평가금액/비중 추이 조회, Yahoo 차트

targets · target_metrics

목표가 수식 평가, 메트릭 보강

time_windows

22시 결산 기준 Today window, intraday baseline

valuation

밸류에이션 보조 계산

runtime_quotes · quote_service ✦

배치/외부 코드가 라우터 private에 의존하지 않게 하는 provider seam 정상화 완료. quote 구현(현재가/현금/외부자산 디스패처/provider)을 quote_service로 이동, _get_provider의 routes 역import 제거. 라우터는 patch용 얇은 위임자만 유지.

stock_quotes (services/)

국내 시세 단일 진입점: WS 캐시 우선 → REST fallback, dead-stock 처리, 지속 구독

6 데이터 · 캐시 계층

SQLite (cache.db, WAL)

주요 테이블 군: 분석 캐시(KV), corp_codes, financial/market_data, users/sessions, user_portfolio, portfolio_cashflows, 스냅샷(nav / stock / group / stock_weight / intraday), NPS, wiki(+FTS), benchmark, system_events, ai_usage_events, app_settings, pdf_cache.

  • 웹·배치가 동일 파일에 동시 쓰기 → 모든 커넥션에 busy_timeout=5000, 다중 statement write는 BEGIN IMMEDIATE 트랜잭션 + rollback
  • 그룹/종목 비중은 스냅샷 시점에 사전 집계(차트 조회를 가볍게 유지)

인메모리 TTL 캐시 (cache_layer.MemoryTTLCache)

시세(stock.current / dead), FX(rates / daily), 벤치마크, insight FX, yfinance 음성 캐시(TTL — 일시적 실패 자동 회복), KIS WS quote 캐시 등.

7 실시간 시세 파이프라인

브라우저 ├─ WebSocket /ws/quotes (Origin 검증) ─▶ kis_ws_manager (KIS WS, 키 슬롯 N개, takeover) │ └─ 실시간 체결 ─▶ _quote_cache ─▶ relay ─▶ 브라우저 └─ REST POST /api/asset-quotes (초기/폴백) ├─ 국내(KRX) : stock_price.fetch_bulk_quotes_kr (네이버 벌크 1회) ✦ │ ↳ 실패 시 종목별 KIS REST fallback ├─ 해외 : KIS 해외 → yfinance → 네이버 world (→ fx 원화 변환) ├─ 금/크립토 : special_assets (네이버/Upbit) └─ 현금성 : fx 환율 공통 경유: services.stock_quotes (WS 캐시 우선 → REST, dead-stock TTL, should_accept 가드)
오늘 개선: 초기 로딩이 종목별 N회 호출 → 국내는 네이버 벌크 1회로. WS는 핸드셰이크 Origin 검증·키 슬롯 누수 차단. 해외는 yfinance 일시 실패가 영구 캐시되던 버그를 TTL 캐시로 자동 회복.

10 남은 기술 부채 · 다음 단계
