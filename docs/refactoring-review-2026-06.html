1. 요약 및 종합 평가

한 줄 결론: 2026-04에 세운 재설계 계획(docs/rearchitecture-plan.md)은 약 60% 진행되어 골격(core / routes / services / repositories)은 자리를 잡았다. 지금 가장 큰 병목은 cache.py가 여전히 19개 repository를 재수출하는 신(神)-모듈 허브로 남아 11개 레거시 모듈이 이를 경유한다는 점이다. 다음 리팩토링 물결은 이 허브를 해체하는 것에서 시작해야 하며, 그 전에 CI의 빈 구멍(JS 테스트 미실행 · lint 부재 · health-check 비차단)부터 막는 것이 비용 대비 효과가 가장 크다.

잘 되어 있는 것 (유지할 것)

  • 서브프로젝트 운영 모델 — 각 대시보드는 독립 배포 유지, 허브는 published JSON·딥링크·설정 관리로만 결합. 코드 복제가 없다.
  • 배포 파이프라인 — push마다 전체 pytest가 배포를 차단하고 실패 시 자동 롤백(deploy/deploy.sh), systemd Type=notify + watchdog + 리소스 캡, 실패 시 ntfy 알림.
  • 백업 — WAL-safe 온라인 백업 + 무결성 검사 + 보존 정책(일 14회·주 60일) + 오프사이트 옵션.
  • 보안 기본기 — 파라미터화 쿼리 일관, HMAC 세션 토큰, 상수시간 비교 내부 토큰, 관리자 mutation CSRF 가드(Origin/Referer), DOMPurify 기반 _renderSafeMarkdown() XSS 방어.
  • 자기 문서화 습관 — health-review, 재설계 계획, NPS 분리 계획, 아키텍처 그래프, 패치노트가 실제 코드와 대체로 일치.
최우선 해결 과제 5건
  1. cache.py(1,440줄) 신-모듈 해체 — 11개 레거시 모듈의 우회 import가 모든 후속 리팩토링을 막고 있다.
  2. CI에 JS 테스트(npm test, 10개 파일 1,373줄)가 빠져 있다 — 로컬에서만 실행되어 JS 회귀가 배포를 통과한다.
  3. 다중 statement 쓰기의 트랜잭션 원자성 부재 — 포트폴리오 항목 저장이 4~5회의 개별 commit으로 수행되어 중간 실패 시 불일치 상태가 가능하다.
  4. 광역 except Exception 231곳 — HTTP 오류·레이트리밋·DB 오류·파싱 오류가 구분되지 않아 운영 디버깅이 어렵다.
  5. 배포 health-check 실패가 배포를 차단하지 않는다 — 테스트는 막지만 기동 실패는 "기록만" 된다.

2. 시스템 전경: 허브와 서브프로젝트

4. 백엔드 구조·품질 평가

4.1 레이어링: 목표 vs 현실

목표 구조(routes → services → repositories → DB)는 신규 코드에서는 지켜지지만, 11개 모듈이 repository를 직접 쓰지 않고 cache를 import한다:

ai_config.py · benchmark_history.py · dart_report_review.py · deps.py · foreign_dividends.py · market_daily.py · observability.py · preferred_dividends.py · snapshot_intraday.py · snapshot_nav.py · wiki_ingestion.py

cache.py 해부 (1,440줄) — 남아 있는 것:
  • DB 연결 싱글턴 get_db() + WAL/busy_timeout/FK pragma (cache.py:18)
  • init_db() 스키마 + 마이그레이션 로직 (cache.py:200~)
  • 비즈니스 로직: 그룹/종목 비중 스냅샷 재계산(_refresh_group_snapshots :38, _refresh_stock_weight_snapshots :118), 현금성/특수자산 판정, DART corp-code 검색
  • 파일 하단 1,260~1,439줄: 순환 import 회피용 늦은 import로 19개 repository에서 ~100개 함수 재수출 — 추출이 "절반에서 멈춘" 명확한 증거

또한 repository들이 거꾸로 cache.get_db()를 호출해(예: repositories/snapshots.py:20) 양방향 결합이 형성돼 있다. 연결 관리를 repositories/db.py 같은 하위 모듈로 내리면 이 순환이 끊긴다.

5.2 프런트 테스트의 구조적 한계

test_frontend_structure.py의 문자열 존재 검사는 "구조 계약"(로드 순서, 파일 크기 상한, 함수 시그니처 존재)을 지키는 데는 효과적이지만, 로직 실행을 검증하지 않고 함수 리네임에 깨지는 brittle한 방식이다. 다행히 tests/js/*.test.mjs(jsdom 기반 행위 테스트 10개)로의 점진 대체가 이미 시작됐다 — 문제는 이것이 CI에서 돌지 않는다는 것(6장).

6.3 운영 (단일 서버 SQLite)

  • 양호 systemd Type=notify + WatchdogSec=30s + MemoryMax 1.5G/CPUQuota 180% + ProtectSystem=strict + OnFailure ntfy 알림. WAL + busy_timeout 5s + FK ON.
  • 양호 백업: 온라인 .backup + gunzip 후 PRAGMA integrity_check + 보존 정책 + 이메일/rclone 오프사이트.
  • 보완 복구 리허설 없음 — 백업은 있는데 restore 절차 문서·DR 드릴이 없다. VACUUM 전략 부재(스냅샷 테이블 무한 성장). 배치 run-record 테이블 부재(계획 6단계).
  • 보완 루트 README.md 부재 — docs/는 풍부하나 진입점이 없다. API 레퍼런스·스키마 ERD·운영 런북(복구/이벤트 조회/메모리 조정) 미비.

7. 서브프로젝트 연동 평가와 방향

7.1 평가

"각 프로젝트는 독립 배포를 유지하고, 허브는 데이터와 내비게이션만 조합한다"는 운영 원칙이 코드에서 실제로 지켜지고 있다 — 이것이 이 멀티 프로젝트 체계의 가장 큰 자산이다. NPS 분리(깨지기 쉬운 스크래퍼를 배포 게이트에서 제거)는 이 원칙의 모범 사례였고, 대시보드 분리는 완료됐다.

리스크내용제언
암묵적 JSON 스키마 계약external_tools.py가 5개 대시보드의 current.json/data.json 구조를 가정하나, 스키마 버전이나 계약 테스트가 없다. 서브프로젝트가 필드를 바꾸면 인사이트가 조용히 빠진다각 published JSON에 schema_version 필드 추가 + 허브에 fixture 기반 계약 테스트(이미 test_external_tools.py 골격 존재 — 실제 스키마 스냅샷으로 보강)
finance-pi 단일 장애점화 진행종가 백업 + (계획상) NPS 백테스트 유니버스 + preferredSpread 내부 API까지 finance-pi 의존이 늘고 있다. 하드코딩 기본값 http://192.168.68.84finance-pi 헬스 상태를 admin 관측성 패널에 추가, 의존 추가 시마다 "다운 시 동작" 명시(현재 종가 백업은 graceful)
연동 상태 가시성각 대시보드 JSON fetch 실패는 독립 격리되지만, 실패가 누적돼도 운영자가 모름/api/integrations에 마지막 성공 fetch 시각·연속 실패 횟수 노출 → admin 패널 표시
kis-proxy 결합도시세의 사실상 전부(REST+WS)가 kis-proxy 경유. 프록시 장애 시 yfinance/Naver 폴백이 모듈마다 제각각4.3의 quote 단일화(services/stock_quotes 확장)와 함께 폴백 정책을 한 곳에 명문화

7.2 서브프로젝트 방향

  • NPS 분리 계획 마무리 — 남은 절반(백테스트 유니버스 finance-pi 이관)을 실행해 nps_holdings 의존을 허브에서 제거. 계획 문서가 이미 상세하므로 실행만 남았다.
  • 신규 서브프로젝트 추가 기준 유지 — "깨지기 쉬운 스크래핑은 허브 밖, 안정 데이터는 허브 안" 기준이 검증됐다. economic_calendar·market_news 같은 스크래핑 의존 모듈도 장기적으로 같은 기준으로 재평가할 가치가 있다.
  • 공통 인프라 문서화 — kis-proxy·finance-pi는 여러 프로젝트가 공유하는 인프라가 됐다. 두 서버의 SLO(가용성 기대치)와 장애 시 각 소비자의 동작을 한 문서로 정리하면 디버깅 시간이 줄어든다.

9. 신규 기능 제언

기준: 이미 보유한 데이터 자산(NAV 스냅샷·배당·공시·AI 파이프라인·finance-pi PIT 데이터)을 재활용하면서, "기다리지 않는 화면"(배치 사전 생성 + 캐시 즉시 표시)이라는 기존 설계 철학에 맞는 것 우선.

제언내용활용 자산난이도
① 리밸런싱 도우미 종목/그룹별 목표 비중을 설정하고, 현재 비중과의 드리프트가 임계치를 넘으면 기존 조건 알림 채널(텔레그램/카카오)로 통지. "몇 주 사고팔면 목표 비중"인지 제안 표시 그룹/종목 비중 스냅샷(이미 사전 계산됨) + notifications 엔진 + 목표가 수식 인프라 낮음
② 포트폴리오 리스크 패널 NAV 히스토리 기반 MDD·변동성·샤프·벤치마크 대비 베타(베타 오버레이 일부 존재)·종목 간 상관 행렬. 일일 스냅샷 시 배치 계산해 캐시 portfolio_snapshots NAV 시계열 + benchmark_daily + 기존 베타 계산 낮음
③ 백테스트 통합 finance-pi PIT 데이터레이크를 유니버스 소스로 하는 룰 기반 백테스트(가치 팩터 스크리닝 → 보유기간 수익률). NPS 분리 계획의 백테스트 이관과 같은 트랙으로 진행 finance-pi Bronze/Silver/Gold + close_price_client + (이관 예정) NPS 시점별 보유내역 높음 — finance-pi 측 작업 병행 필요
④ 배당 캘린더 & 세후 수익률 보유 종목의 배당락/지급일 캘린더 뷰(경제캘린더 UI 재활용), 월별 예상 배당 현금흐름, 배당소득세 반영 세후 수익률 국내(DART)·외국·우선주 배당 데이터 3종 이미 수집 중 + economic-calendar.js UI 패턴 중간
⑤ AI 데일리 브리핑 푸시 아침 배치로 "어제 내 포트폴리오: 기여도 상위/하위, 신규 공시·리포트 요약, 오늘 일정"을 사전 생성해 알림 채널로 발송. 생성형 대기시간 0의 기존 철학 그대로 AI 파이프라인(모델 레지스트리·사용량 ledger) + DART 리뷰 + wiki + notify-* systemd 패턴 중간
⑥ 투자 일지 (결정 기록) 매수/매도 시점에 "왜"를 기록하고, 이후 수익률·당시 목표가와 함께 회고하는 뷰. 가치투자 도구로서 차별화 포인트이며 현금흐름 기록과 자연 결합 cashflows 테이블 + 종목 분석 화면 낮음
⑦ PWA + 웹푸시 manifest + service worker로 홈화면 설치, 텔레그램/카카오 외 네이티브 웹푸시 채널 추가. 모바일 simple mode가 이미 있어 효과 즉시 기존 모바일 최적화 + notifications 엔진의 채널 추상화 중간
⑧ 데이터 품질 모니터 scripts/check_data_quality.py를 정기 배치로 승격: 스크래핑 소스(Naver·yfinance)별 최근 성공률·값 급변 감지를 system_events에 기록, admin 패널 + 임계 초과 시 알림. 스크래핑 fragility(6장)의 운영적 대응 기존 스크립트 + observability 인프라 + systemd timer 패턴 낮음
권장 착수 순서: ②리스크 패널 → ①리밸런싱 → ⑧데이터 품질 모니터 (셋 다 기존 데이터+알림 인프라 재활용, 신규 외부 의존 0) → ④배당 캘린더 → ⑤AI 브리핑 → ⑥투자 일지 → ⑦PWA → ③백테스트. 단, Phase 1(cache.py 해체)이 끝나기 전에는 신규 기능이 레거시 import 패턴을 더 쌓지 않도록 — 신규 코드는 반드시 services/repositories 경유 원칙을 지킬 것.

10. 결론

이 프로젝트는 개인 운영 규모에서 보기 드물게 "계획 → 집행 → 문서화" 사이클이 살아 있는 코드베이스다. 2026-04 재설계 계획의 절반 이상이 실제로 집행됐고, 배포·백업·보안 기본기는 이미 상위권이다. 남은 과제는 명확하다:

  1. 안전망의 구멍 3개를 먼저 막는다 — JS 테스트 CI 편입, health-check 차단, ruff. 모두 합쳐 며칠 거리다.
  2. cache.py 허브를 해체한다 — 이것이 끝나야 시세 단일화·트랜잭션·예외 체계화가 풀린다. 완료 판정 기준은 단순하다: cache.py 하단의 재수출 블록을 지울 수 있는가.
  3. 신규 기능은 기존 데이터 자산 재활용형부터 — 리스크 패널·리밸런싱·데이터 품질 모니터는 새 외부 의존 없이 즉시 가치를 낸다.
  4. 서브프로젝트 경계 원칙은 그대로 유지하되, JSON 스키마 계약 명시와 연동 상태 가시성만 보강한다.

방법론과 한계 — 본 보고서는 2026-06-10 기준 master(84d42a6) 워킹트리에 대한 정적 분석(전수 구조 스캔 + 핵심 파일 정독 + 메트릭 실측)과 저장소 내 설계 문서 대조로 작성됐다. 수치(LOC·예외 핸들러 수·innerHTML 수 등)는 스크립트로 실측한 값이다. 서브프로젝트 7개 저장소의 내부 코드는 이 세션에서 접근할 수 없어, 해당 부분은 허브 측 연동 코드와 문서를 근거로 한 평가임을 밝힌다. 런타임 프로파일링(실서버 응답시간·메모리)은 범위에 포함되지 않았다.
