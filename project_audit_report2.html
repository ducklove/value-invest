1프로젝트 요약

현재 상태·포지셔닝·총평

Value Compass는 개인 가치투자자를 위한 포트폴리오·종목분석 허브다. FastAPI 단일 서버가 REST API·정적 SPA·관리자 콘솔·KIS 실시간 시세 WebSocket·내부 배치 트리거를 함께 제공하고, SQLite 단일 파일(cache.db, 46개 테이블)에 분석 캐시·사용자·포트폴리오·NAV 스냅샷·AI 사용량·공시/리포트 요약을 저장한다. 운영은 라즈베리파이 + systemd 타이머, 배포는 self-hosted GitHub Actions가 deploy/deploy.sh(ruff→pytest→JS 3중 게이트 + 실패 시 롤백 + healthz 재검)로 수행한다.

구조는 main.py → core/app_factory.py 조립을 중심으로 routes/(24개)·services/(portfolio 23개 등)·repositories/(26개)로 계층화가 진행 중이며, 루트에는 아직 31개 레거시 모듈(13.6K LOC, 백엔드의 36%)이 이관 대기 상태로 남아 있다. AI 계층(포트폴리오 인사이트 스트리밍·증권사 리포트 위키 요약·DART 정기보고서 리뷰·아침/마감 브리핑)이 기능별 모델 레지스트리와 사용량 예산 캡 위에서 이미 폭넓게 동작한다.

총평. 이 저장소는 "리아키텍처가 절반쯤 진행된, 규율 있는 1인 운영 제품"의 전형이다. 좋은 공용 인프라(repositories 계층·cache_layer TTL 캐시·core/http 공유 클라이언트·구조 경계 테스트)가 설계·문서화까지 되어 있으나 채택이 미완이고, 방향(재설계 계획·기능 확장)은 분명하나 이행이 중간 지점에 멈춰 있다. 개선의 핵심은 새 설계가 아니라 이미 만들어 둔 추상화로의 "채택 완주"와 몇 개의 뚜렷한 구멍(반사형 XSS·레이트리밋 부재·PR CI 부재·침묵 실패) 메우기다.

async function apiFetchJson(path, options = {}) {
  const resp = await apiFetch(path, options);
  if (!resp.ok) {
    const d = await resp.json().catch(() => ({}));
    const err = new Error(d.detail || `HTTP ${resp.status}`);
    err.status = resp.status;
    throw err;
  }
  return resp.json();
}

3. 종목 리서치 메모리 — 위키·DART 리뷰·일지를 횡단하는 RAG Q&A
문제 정의AI가 만든 리서치 자산이 세 개의 고립된 저장소에 쌓인다: 증권사 리포트 위키(wiki_ingestion.py:1-9 파이프라인 → stock_wiki_entries), DART 정기보고서 리뷰(dart_report_review.py:1 "AI review pipeline for the latest DART periodic filing"), 투자 일지(repositories/journal.py). 그러나 포트폴리오 AI 인사이트는 종목당 최근 위키 3건·15종목까지만 잘라 넣는 고정 주입(services/portfolio/ai_analysis.py:50-52 WIKI_HOLDING_LIMIT=15, WIKI_ENTRY_LIMIT=3, WIKI_KEYPOINT_CHARS=600)이라 축적 데이터의 극히 일부만 활용되고, "삼성전자에 대해 지난 1년간 리포트들이 뭐라고 했지?" 같은 횡단 질문에 답할 수 없다.
사용자 가치종목·포트폴리오 질문에 대해 '내가 모은 리서치 전체'(리포트 요약 수백 건 + 공시 리뷰 + 내 판단 기록)를 근거로, 출처 링크가 달린 답을 받는다. 시간이 갈수록 내 데이터로 더 똑똑해지는 개인 리서치 어시스턴트.
비즈니스 가치이 제품의 유일무이한 해자는 사용자가 수년간 축적한 리서치 코퍼스다. 이를 검색·회수 가능하게 만들면 데이터가 쌓일수록 경쟁 제품으로 못 떠나는 구조가 된다.
필요 기술SQLite FTS5(표준 SQLite 내장 — 신규 의존성 없음), 한국어 토크나이즈는 FTS5 trigram으로 시작. 임베딩은 확장 단계에서만(sqlite-vec 또는 finance-pi 위임) 검토

문제 정의인프라는 이미 멀티유저다: Google OAuth 세션, 모든 저장소의 google_sub 스코핑(예: repositories/journal.py:8-10 "소유권은 모든 쿼리에 google_sub 조건"), 사용자별 알림 자격증명, 사용자·사이트 이중 AI 예산 캡(ai_config.py:400-421 enforce_budget_caps, services/ai_client.py:110-114에서 매 호출 강제), 인사이트 보드의 public/private(routes/insights.py:24). 그러나 신규 사용자의 온보딩 경로·역할 관리·성과 공유 수단이 없어, 사실상 운영자 1인 제품에 머물러 있고 축적된 기능이 외부 가치로 전환되지 않는다.
